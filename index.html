Bridging the Protocol Divide

Classic IP cameras output RTSP, a protocol web browsers explicitly reject. Converting industrial video for web consumption requires a strictly governed translation bridge, unpacking legacy streaming formats and repacking them into low-latency WebRTC streams under harsh network constraints.

[ L1 Base ] The Incompatibility Problem

Millions of deployed IP cameras and industrial visual sensors speak exactly one robust language: the Real Time Streaming Protocol (RTSP). It is ancient, UDP-or-TCP flexible, and universally supported by heavy Video Management Systems. Modern web browsers, however, live in a tightly sandboxed reality. Browsers do not speak RTSP. They cannot open arbitrary TCP sockets to pull a stream, and they certainly cannot ingest raw RTSP payloads over a standard <video> tag without massive latency penalties via intermediate formats like Apple's HLS.

If the objective is sub-second latency from a warehouse ceiling camera directly into a browser-based operator dashboard, the standard HTTP-based buffering protocols (HLS or DASH) fail entirely. The latency floor for HLS is typically several seconds. The only viable path to real-time ingestion in the browser is WebRTC.

This creates a strict architectural necessity: a bridge layer that sits on the server, ingests the RTSP feed, maintains connection state, repackages the media payloads, and acts as a WebRTC peer to the connecting browsers. The process is not a mere format wrapper. It is an active protocol translation.

[ L2 Ingest ] RTSP Ingest & GStreamer Processing

The first stage of the bridge connects to the camera's RTSP endpoint. We use GStreamer as the engine for this translation because of its pipeline-oriented processing blocks. When the server requests the RTSP stream, the camera typically delivers an H.264 encoded video feed interleaved over RTP (Real-time Transport Protocol). Ingesting this stream cleanly requires decoupling the video frames from the older RTSP presentation layer without degrading the H.264 compression.

Latency preservation rule: If the camera natively encodes H.264, the pipeline must strictly avoid transcoding the video. Transcoding a 1080p stream from H.264 to VP8 on a bridge server adds devastating CPU load and roughly 300–500ms of latency per frame. The pipeline must demux the RTSP container, parse the raw H.264 Network Abstraction Layer (NAL) units, and parse them directly into the WebRTC transport mechanism.

[ L3 Translate ] Delivery & Relay Behavior

Handing the parsed frames over to the browser involves adopting WebRTC's formidable security and session management constraints. webrtcbin in GStreamer handles the complex work of Datagram Transport Layer Security (DTLS) handshakes and packetizing the NAL units into Secure Real-Time Transport Protocol (SRTP) packets.

However, an edge server dropping SRTP packets onto the internet is meaningless if the client cannot route them. Since the server bridge acts as an authoritative peer, it must negotiate ICE (Interactive Connectivity Establishment) candidates with the connecting browser. The bridge server advertises its own public IP through STUN. The browser receives this IP via the signaling channel (usually WebSockets) and attempts to connect.

// Example GStreamer pipeline segment establishing the WebRTC delivery
pipeline_str = 
  "rtspsrc location=rtsp://cam/stream latency=100 ! "
  "rtph264depay ! h264parse ! "
  "rtph264pay config-interval=-1 ! "
  "webrtcbin name=sendrecv bundle-policy=max-bundle";

When multiple browsers request the same camera feed, the architecture faces a fan-out limitation. WebRTC is designed as a peer-to-peer specification. The server must establish a unique, encrypted SRTP session with every single connected browser. A single camera stream fanning out to ten dashboard operators requires the bridge server to encrypt and push ten independent media streams simultaneously. Consequently, network outbound capacity on the bridge block is usually the hard bottleneck before CPU hits limitations.

[ L4 Record ] Sidecar Archiving & Load Boundaries

The secondary requirement for industrial vision is auditing, which requires persistent recording. In a typical VMS, the server manages the recording directly from the RTSP payload. We mirror this in the bridge using a tee element in the GStreamer pipeline.

The tee splits the H.264 stream. While one branch pushes packets to webrtcbin, the secondary branch pushes to a heavy mp4mux or matroskamux element, writing to disk in rolling 15-minute segments. Because this fork occurs after the H.264 stream is extracted from RTP, the disk I/O requires zero additional encoding overhead.

Ultimately, a successful RTSP-to-WebRTC bridge is defined by its parsimony. It strips the wrapper, avoids touching the pixels, negotiates the tunnel, and secures the delivery. Anything more is a waste of cycles.
